## ⚡ 프로젝트 4: 실시간 이벤트 로그 분석 파이프라인

### 프로젝트 개요

| 항목 | 내용 |
|------|------|
| **프로젝트명** | StreamPipe: 실시간 이벤트 로그 분석 파이프라인 |
| **한줄 요약** | "PySpark + Kafka로 일 1억 건 클릭스트림 처리 — Batch vs Streaming 성능 비교 + 실시간 이상 탐지" |
| **데이터** | NYC Taxi Trip Records (월 1,300만+ 건) + 시뮬레이션 클릭스트림 |
| **기술 스택** | PySpark, Kafka, Delta Lake, Docker, Streamlit |
| **배포** | Streamlit (대시보드) + Docker Compose (파이프라인) |
| **글로우색상** | amber (자동화) |

### 왜 NYC Taxi인가

- **공식 데이터, 즉시 접근 가능** (NYC TLC 공개)
- 월 1,300만 건 × 12개월 = **연 1.5억 건** → "대규모" 증명
- 위치, 시간, 금액, 팁 등 분석 가능한 피처 풍부
- 클릭스트림 시뮬레이션과 병행하여 다양한 데이터 유형 경험

### Day-by-Day 플랜

#### Day 1: 환경 설정 + 데이터 준비 (4~5h)

```
📋 할 일
├── Docker 환경 구성
│   ├── docker-compose.yml (Spark, Kafka, Zookeeper)
│   ├── PySpark 로컬 설치 + 설정 확인
│   └── Kafka 브로커 + 토픽 생성
├── 데이터 준비
│   ├── NYC Taxi 2023 데이터 다운로드 (Parquet 형식)
│   ├── 데이터 크기 확인 (행 수, 파일 크기)
│   ├── 스키마 파악 + 필요 컬럼 선별
│   └── 클릭스트림 시뮬레이터 작성
│       ├── user_id, session_id, event_type, timestamp, page, product_id
│       ├── 현실적 패턴 (시간대별 트래픽, 세션 길이, 이탈률)
│       └── Kafka Producer로 스트리밍 전송
├── 기본 PySpark 연결 테스트
│   ├── Parquet 읽기
│   ├── 기본 집계 (행 수, 컬럼별 통계)
│   └── Spark UI 접근 확인
└── 산출물: docker-compose.yml, clickstream_generator.py
```

#### Day 2: Batch 처리 파이프라인 (4~5h)

```
📋 할 일
├── NYC Taxi Batch 분석 (PySpark)
│   ├── 데이터 품질 검사 (null, 이상치, 범위 초과)
│   ├── 시간대별 승차 패턴 분석
│   ├── 지역별 매출 집계 (Zone 기반)
│   ├── 팁 비율 분석 (결제 방식별)
│   ├── 수요 예측용 Feature 생성
│   └── 결과를 Delta Lake / Parquet로 저장
├── 파이프라인 최적화
│   ├── 파티셔닝 전략 (날짜별, 지역별)
│   ├── 파티셔닝 전후 쿼리 성능 비교 (실행 시간 측정)
│   ├── 캐싱 효과 측정
│   ├── Broadcast Join vs Shuffle Join 비교
│   └── Spark UI에서 Stage/Task 분석
├── 성능 벤치마크 기록
│   ├── pandas vs PySpark 처리 시간 비교 (동일 작업)
│   ├── 데이터 크기별 스케일링 테스트 (100만/1000만/전체)
│   └── 메모리 사용량 비교
└── 산출물: 01_batch_pipeline.ipynb, src/batch/
```

#### Day 3: Streaming 파이프라인 (4~5h)

```
📋 할 일
├── Kafka → Spark Structured Streaming
│   ├── 클릭스트림 시뮬레이터 → Kafka Topic
│   ├── Spark readStream으로 Kafka 소비
│   ├── Watermark 설정 (late data 처리)
│   └── Window 기반 집계 (5분/15분/1시간)
├── 실시간 분석
│   ├── 실시간 DAU/MAU 카운트
│   ├── 실시간 전환 퍼널 (page_view → cart → purchase)
│   ├── 실시간 이상 탐지 (트래픽 급증/급감)
│   │   ├── Z-score 기반 (최근 1시간 평균 대비)
│   │   └── 이상 감지 시 알림 로직
│   └── 결과를 메모리/파일 싱크로 출력
├── Batch vs Streaming 비교
│   ├── 동일 분석을 Batch와 Streaming으로 구현
│   ├── 지연 시간(Latency) 비교
│   ├── 처리량(Throughput) 비교
│   └── "언제 Batch, 언제 Streaming?" 의사결정 가이드
└── 산출물: 02_streaming_pipeline.ipynb, src/streaming/
```

#### Day 4: Delta Lake + 데이터 품질 (4~5h)

```
📋 할 일
├── Delta Lake 도입
│   ├── Parquet → Delta Lake 마이그레이션
│   ├── ACID 트랜잭션 데모
│   ├── Time Travel (버전 관리) 데모
│   ├── Schema Evolution (스키마 변경 대응) 데모
│   └── OPTIMIZE + ZORDER (쿼리 최적화)
├── 데이터 품질 파이프라인
│   ├── Great Expectations 또는 커스텀 DQ 규칙
│   ├── null 비율 체크
│   ├── 범위 검증 (음수 요금, 미래 날짜)
│   ├── 중복 레코드 탐지
│   └── DQ 결과 리포트 생성
├── 파이프라인 아키텍처 문서
│   ├── 아키텍처 다이어그램 (Source → Kafka → Spark → Delta → Dashboard)
│   ├── 각 컴포넌트 역할 설명
│   └── 확장 전략 (Worker 추가, 파티션 증가)
└── 산출물: 03_delta_lake.ipynb, src/quality/
```
